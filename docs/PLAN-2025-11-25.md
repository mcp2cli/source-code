# Implementation Plan: MCP `2025-11-25` Compliance

> **Date:** 2026-03-28
> **Scope:** Bring mcp2cli from protocol version `2025-03-26` to `2025-11-25`.
> **Approach:** Each work item is decomposed into exact file changes, with full
> awareness of how our architecture dispatches operations.

---

## Architecture Recap (for context)

The request lifecycle flows through these layers:

```
CLI command  (apps/dynamic.rs)
  → McpOperation enum  (mcp/model.rs)
  → map_operation_to_request()  (mcp/protocol.rs)
  → transport.send()  (mcp/client.rs — Stdio or StreamableHttp)
  → JsonRpcResponse
  → map_streamable_http_response()  (mcp/client.rs — shared mapper)
  → McpOperationResult enum  (mcp/model.rs)
  → format_*_result()  (apps/dynamic.rs)
  → CommandOutput  (output/mod.rs)
```

Server→client messages during a pending request arrive at
`OperationMessageHandler` (mcp/handler.rs), which dispatches notifications
and handles `elicitation/create` and `sampling/createMessage`.

Discovery results flow into `DiscoveryInventoryView` (runtime/state.rs), then
into `CommandManifest` (apps/manifest.rs), which builds the clap CLI surface.

## Work Items

### WI-01: Structured Content + `outputSchema`

**What the spec says:** Tools may declare an `outputSchema` in `tools/list` and
return typed data in `structuredContent` alongside the existing `content` array.
"Clients SHOULD validate structured results against this schema."

**Why it matters for our CLI:** We are a *structured output tool* — `--json` mode
should emit `structuredContent` as first-class data, not a stringified summary.
And `ls --detail` should show the output schema so users know what shape to parse.

**Implementation:**

1. **Discovery layer** — `map_discovery_response()` in [client.rs](../src/mcp/client.rs)
   already extracts `inputSchema` and `annotations` from tools. Add:
   ```rust
   if let Some(output_schema) = tool.get("outputSchema") {
       item["outputSchema"] = output_schema.clone();
   }
   ```
   This preserves the schema in the inventory item for manifest building and
   detailed display.

2. **Manifest layer** — In `tool_to_command()` in [manifest.rs](../src/apps/manifest.rs),
   no change needed — the manifest doesn't need to know about output schemas
   (they don't affect CLI flags). But we should propagate it for display.

3. **Result mapper** — `map_tool_call_response()` in [client.rs](../src/mcp/client.rs)
   currently calls `tool_call_summary()` which checks
   `result.get("structuredContent").is_some()` but falls back to a label.
   Change `tool_call_summary()` to:
   ```rust
   fn tool_call_summary(capability: &str, result: &Value) -> String {
       // Prefer structuredContent for summary
       if let Some(sc) = result.get("structuredContent") {
           return serde_json::to_string_pretty(sc)
               .unwrap_or_else(|_| format!("{} returned structured content", capability));
       }
       // Fall back to text content blocks
       let content = result.get("content").and_then(Value::as_array)...;
       // ...existing logic...
   }
   ```

4. **Output rendering** — `format_action_result()` in [dynamic.rs](../src/apps/dynamic.rs)
   reads `McpOperationResult::Action { data, .. }`. The `data.result` field
   already contains the raw server response including `structuredContent`.
   In human mode, we should do:
   - If `structuredContent` exists, pretty-print it as JSON.
   - If not, show text content lines (existing behavior).
   In JSON mode, `data.result.structuredContent` is already included (it's the
   raw response).

5. **`ls --detail` for tools** — When rendering tool details in `execute_ls()`,
   show `outputSchema` if present:
   ```
   TOOLS:
     get_weather              Get current weather
       output: { temperature: number, conditions: string, humidity: number }
   ```

6. **Validation (SHOULD)** — Add a `validate_structured_content()` helper that
   checks `structuredContent` against `outputSchema` using basic JSON Schema
   checks (type, required fields). Log a warning if validation fails but still
   return the result. We can use `jsonschema` crate or a minimal inline checker.
   **Decision:** Defer full jsonschema validation to a later phase. For now, log
   a debug warning if types don't match.

**Files touched:**
- `mcp/client.rs` — discovery mapper, tool_call_summary, map_tool_call_response
- `apps/dynamic.rs` — format_action_result, execute_ls

**Effort:** Small-medium.

### WI-02: `resource_link` Content Type

**What the spec says:** Tool results can include `{ "type": "resource_link",
"uri": "...", "name": "...", "description": "...", "mimeType": "..." }`.
"Resource links returned by tools are not guaranteed to appear in
`resources/list`."

**Why it matters for our CLI:** A tool might return a reference to a file or
API resource. If we ignore it, the user loses actionable information.

**Implementation:**

1. **`tool_call_summary()`** — After checking for text content, also check for
   `resource_link` items:
   ```rust
   let links: Vec<String> = content.iter()
       .filter(|item| item.get("type").and_then(Value::as_str) == Some("resource_link"))
       .filter_map(|item| item.get("uri").and_then(Value::as_str).map(ToOwned::to_owned))
       .collect();
   ```
   Append them to the summary: `"[link: file:///project/src/main.rs]"`.

2. **Human rendering in `format_action_result()`** — When iterating content items
   for display, add a branch for `resource_link`:
   ```
   → main.rs (file:///project/src/main.rs) [text/x-rust]
   ```

3. **`--follow-links` flag** — Optional: when present, automatically call
   `resources/read` for each `resource_link` and include the content inline.
   **Decision:** Defer to future. Just display the links for now.

4. **Also handle other content types in the same pass:**
   - `type: "image"` → `[image: <mimeType>, <length> bytes (base64)]`
   - `type: "audio"` → `[audio: <mimeType>, <length> bytes (base64)]`
   - `type: "resource"` → Show embedded resource URI and text preview.
   This completes WI-18 (rich content rendering) at the same time.

**Files touched:**
- `mcp/client.rs` — tool_call_summary
- `apps/dynamic.rs` — format_action_result (new content type branches)

**Effort:** Small.

### WI-03: URL Mode Elicitation

**What the spec says:** Elicitation now supports two modes:
- `"form"` (existing) — collect structured data via prompts.
- `"url"` (new) — direct user to an external URL for out-of-band interaction.

The capability declaration changes from `elicitation: {}` to
`elicitation: { form: {}, url: {} }`. For backwards compat, `{}` means form-only.

Security: "MUST NOT automatically pre-fetch the URL", "MUST show the full URL",
"MUST NOT open the URL without explicit consent."

**Why it matters for our CLI:** Servers that integrate with third-party OAuth or
payment flows NEED URL mode. Without it, those flows just fail silently.

**Implementation:**

1. **Capability declaration** — In `ProtocolEngine::new()` in
   [protocol.rs](../src/mcp/protocol.rs), change `ClientCapabilities`:
   ```rust
   pub struct ElicitationCapability {
       #[serde(skip_serializing_if = "Option::is_none")]
       pub form: Option<CapabilityMarker>,
       #[serde(skip_serializing_if = "Option::is_none")]
       pub url: Option<CapabilityMarker>,
   }
   ```
   The `ClientCapabilities.elicitation` field changes from `Option<CapabilityMarker>`
   to `Option<ElicitationCapability>`. In `ProtocolEngine::new()`:
   ```rust
   elicitation: Some(ElicitationCapability {
       form: Some(CapabilityMarker {}),
       url: Some(CapabilityMarker {}),
   }),
   ```

2. **Handler dispatch** — In `handle_elicitation_request()` in
   [handler.rs](../src/mcp/handler.rs), check `params.mode`:
   ```rust
   let mode = params.get("mode")
       .and_then(Value::as_str)
       .unwrap_or("form");  // default is form
   
   match mode {
       "form" | "" => handle_form_elicitation(request, params),
       "url" => handle_url_elicitation(request, params),
       other => // return -32602 invalid params
   }
   ```

3. **URL mode handler** — New function `handle_url_elicitation()`:
   ```rust
   fn handle_url_elicitation(request: &JsonRpcRequest, params: &Value) -> Result<JsonRpcResponse> {
       let message = params.get("message").and_then(Value::as_str).unwrap_or("...");
       let url = params.get("url").and_then(Value::as_str)
           .ok_or_else(|| anyhow!("url mode elicitation missing 'url' field"))?;
       let elicitation_id = params.get("elicitationId").and_then(Value::as_str);
   
       eprintln!("--- url elicitation request ---");
       eprintln!("{}", message);
       eprintln!();
       eprintln!("The server asks you to open this URL:");
       eprintln!("  {}", url);
       eprintln!();
       eprint!("Open in browser? [Y/n/cancel]: ");
       io::stderr().flush().ok();
   
       // Read user choice
       let mut line = String::new();
       io::stdin().lock().read_line(&mut line)?;
       let choice = line.trim().to_ascii_lowercase();
   
       let action = match choice.as_str() {
           "" | "y" | "yes" => {
               // Attempt to open browser
               if let Err(e) = open_url_in_browser(url) {
                   eprintln!("Could not open browser: {}. Please open manually.", e);
               }
               eprint!("Press Enter after completing the interaction... ");
               io::stderr().flush().ok();
               let mut wait = String::new();
               io::stdin().lock().read_line(&mut wait)?;
               "accept"
           }
           "n" | "no" => "decline",
           _ => "cancel",
       };
   
       eprintln!("--- end url elicitation ---");
       Ok(json_response(request.id.clone(), action, None))
   }
   ```
   The `open_url_in_browser()` helper uses `open` (macOS), `xdg-open` (Linux),
   or `start` (Windows).

4. **`notifications/elicitation/complete` handling** — In
   `OperationMessageHandler::handle_notification()`, add a case:
   ```rust
   "notifications/elicitation/complete" => self.handle_elicitation_complete(params),
   ```
   This emits a RuntimeEvent::Info so the user sees "elicitation completed" in
   the terminal.

5. **`URLElicitationRequiredError` (code -32042)** — In `into_result()` on
   `JsonRpcResponse` in [protocol.rs](../src/mcp/protocol.rs), we currently
   return a generic error. We should detect error code `-32042` and extract the
   embedded elicitations:
   - Display each URL to the user.
   - After the user completes them, retry the original request.
   **Decision:** Implement the detection and display. Automatic retry is complex
   (needs storing the original request in the client). For now, show the URLs
   and tell the user to retry manually. Mark as TODO for auto-retry.

6. **Non-interactive mode** — If `--non-interactive` is set, all elicitations
   (form and URL) should return `action: "cancel"` without prompting. The
   handler needs access to this flag.
   
   **Design decision:** Pass a `non_interactive: bool` field into
   `OperationMessageHandler`. The handler checks it at the top of both form and
   URL handlers. This requires adding the field to the struct and threading it
   through `McpClient::perform()` → but that's invasive. Alternative: use an
   `Arc<AtomicBool>` on the struct. Or: check an environment variable
   `MCP2CLI_NON_INTERACTIVE=1`. **Go with the struct field** since we already
   create `OperationMessageHandler` at the call site.

**Files touched:**
- `mcp/protocol.rs` — `ClientCapabilities`, `ElicitationCapability` struct
- `mcp/handler.rs` — mode dispatch, URL handler, non-interactive, elicitation complete notification
- `mcp/client.rs` — thread `non_interactive` into handler creation

**Effort:** Medium.

### WI-05: Protocol Version Bump to `2025-11-25`

**What the spec says:** "The client MUST send a protocol version it supports.
This SHOULD be the latest version supported by the client." Version negotiation
allows the server to respond with an older version it supports.

**Implementation:**

1. **Constant change** — In [protocol.rs](../src/mcp/protocol.rs) line 7:
   ```rust
   pub const DEFAULT_MCP_PROTOCOL_VERSION: &str = "2025-11-25";
   ```

2. **Version negotiation handling** — In `complete_initialize()`, we already
   accept whatever version the server responds with:
   ```rust
   session.protocol_version = result.protocol_version;
   ```
   This means if the server responds `"2025-03-26"`, we'll use that. Good.
   
   However, we should also store the negotiated version and use it to
   **feature-gate** behavior. Add a helper:
   ```rust
   impl McpClientSession {
       pub fn supports_version(&self, version: &str) -> bool {
           self.protocol_version.as_str() >= version
       }
   }
   ```
   Then in response mapping, check `session.supports_version("2025-06-18")`
   before trying to parse `structuredContent`, etc.

3. **Dependency:** This change should be the LAST thing in Phase 1 — only bump
   after WI-01 through WI-04 and WI-06 are complete.

**Files touched:**
- `mcp/protocol.rs` — constant, helper method

**Effort:** Trivial (after prerequisites).

### WI-07: Task System (Experimental) — ✅ IMPLEMENTED

**What the spec says:** Tasks are "durable state machines" that wrap requests.
A client augments a `tools/call` with `task: { ttl: 60000 }`, the server returns
a `CreateTaskResult` with a `taskId` instead of the actual result. The client
then polls `tasks/get` and retrieves the final result via `tasks/result`.

Tools declare `execution.taskSupport` as `"forbidden"` (default), `"optional"`,
or `"required"`.

**Why it matters for our CLI:** This is the *async job model*. Our existing
bridge layer has `JobRecord`, `TaskAction`, `TaskState` — all CLI-side concepts
that don't speak MCP protocol. The spec now standardizes this.

**Design decisions for CLI:**

- **When to use tasks:** If a tool has `taskSupport: "required"`, we MUST use
  tasks. If `"optional"`, we use tasks for `--background` invocations. If
  `"forbidden"` (default), never.
- **Polling UX:** The CLI shows a spinner/progress while polling. The terminal
  shows `[working]  get_weather ...` and updates from
  `notifications/tasks/status`.
- **`jobs` command family** — Our existing `jobs list/show/wait/cancel` commands
  map naturally to `tasks/list`, `tasks/get`/`tasks/result`, `tasks/cancel`.
- **Background mode** — `tool --background` uses task augmentation, stores the
  `taskId` in `StateStore.jobs`, and returns immediately. `jobs wait` calls
  `tasks/result` (which blocks).

**Implementation:**

1. **Discovery: parse `execution.taskSupport`** — In `map_discovery_response()`
   for tools, extract:
   ```rust
   if let Some(execution) = tool.get("execution") {
       item["execution"] = execution.clone();
   }
   ```
   In manifest.rs, read `execution.taskSupport` to decide whether to add
   `--background` flag. Currently `supports_background: true` for all tools.
   Change to check the field.

2. **New McpOperation variants:**
   ```rust
   McpOperation::TaskGet { task_id: String },
   McpOperation::TaskResult { task_id: String },
   McpOperation::TaskList { cursor: Option<String> },
   McpOperation::TaskCancel { task_id: String },
   ```

3. **Protocol mapping** — In `map_operation_to_request()`:
   - `TaskGet` → `tasks/get` with `{ taskId }`
   - `TaskResult` → `tasks/result` with `{ taskId }`
   - `TaskList` → `tasks/list` with optional `{ cursor }`
   - `TaskCancel` → `tasks/cancel` with `{ taskId }`

4. **Task-augmented tool call** — When invoking a tool that requires/supports
   tasks, modify the `InvokeAction` mapping:
   ```rust
   McpOperation::InvokeAction { capability, arguments, background } => {
       let mut params = json!({
           "name": capability,
           "arguments": arguments,
       });
       if background {
           params["task"] = json!({ "ttl": 300000 }); // 5 minutes default
       }
       Ok(JsonRpcRequest::new(id, "tools/call", Some(params)))
   }
   ```

5. **Response handling** — A task-augmented response has `result.task` instead of
   `result.content`. In `map_tool_call_response()`:
   ```rust
   if let Some(task_data) = result.get("task") {
       let task_id = task_data.get("taskId").and_then(Value::as_str)...;
       let status = task_data.get("status").and_then(Value::as_str)...;
       return Ok(McpOperationResult::TaskAccepted { ... });
   }
   ```

6. **Polling loop** — For foreground task calls (not `--background`), after
   receiving `TaskAccepted`, immediately enter a poll loop:
   ```rust
   loop {
       sleep(Duration::from_millis(poll_interval)).await;
       let status = client.perform(TaskGet { task_id }).await?;
       match status {
           completed/failed/cancelled => break,
           working => continue (show progress),
           input_required => call TaskResult to get pending request,
       }
   }
   let final_result = client.perform(TaskResult { task_id }).await?;
   ```
   This belongs in `execute_manifest_command()` in dynamic.rs.

7. **`jobs` command wiring** — Existing `DynamicCommand::JobsList` etc. currently
   delegate to bridge. Wire them to the new `McpOperation::Task*` variants
   instead. `jobs list` → `TaskList`, `jobs show <id>` → `TaskGet`,
   `jobs wait <id>` → `TaskResult`, `jobs cancel <id>` → `TaskCancel`.

8. **Client capabilities** — In `ProtocolEngine::new()`, don't declare task
   capabilities on the CLIENT side yet (tasks are server-side for tool calls).
   The client only needs to check `server_capabilities.tasks.requests.tools.call`
   to know whether task-augmented tool calls are supported.

9. **`notifications/tasks/status`** — Add handler in
   `OperationMessageHandler::handle_notification()`.

10. **Feature gate** — Behind `--experimental-tasks` flag or config option until
    the spec stabilizes. Check the flag before augmenting tool calls with tasks.

**Files touched:**
- `mcp/model.rs` — new McpOperation/McpOperationResult variants
- `mcp/protocol.rs` — map_operation_to_request, ServerCapabilities (add tasks)
- `mcp/client.rs` — map_tool_call_response (task detection), map for task operations
- `mcp/handler.rs` — notifications/tasks/status
- `apps/dynamic.rs` — foreground poll loop, jobs command wiring
- `apps/manifest.rs` — supports_background based on execution.taskSupport
- `runtime/state.rs` — store task IDs in job records

**Effort:** Large. This is the biggest work item.

### WI-08: Cancellation Protocol (`notifications/cancelled`) — ✅ IMPLEMENTED

**What the spec says:** "When a party wants to cancel an in-progress request, it
sends a `notifications/cancelled` notification with `{ requestId, reason? }`."

**Why it matters for our CLI:** When the user presses Ctrl+C during a long tool
call, we should notify the server so it can abort work. Currently we just kill
the connection.

**Implementation:**

1. **Track in-flight request IDs** — In `StdioMcpClient` and
   `StreamableHttpMcpClient`, store the current request ID in an
   `Arc<Mutex<Option<JsonRpcId>>>`. Set it before sending, clear after receiving.

2. **Ctrl+C handler** — Register a `tokio::signal::ctrl_c()` handler that:
   a. Reads the current in-flight request ID.
   b. Sends `notifications/cancelled` with `{ requestId: <id> }`.
   c. Then proceeds with normal shutdown.

3. **Stdio transport** — Send the notification as a JSON line to stdin.
4. **HTTP transport** — Send a POST with the notification body.

5. **Server→client cancellation** — Also handle the server sending us
   `notifications/cancelled` for our server→client requests (elicitation,
   sampling). In `handle_notification()`, add the case and log it.

**Files touched:**
- `mcp/client.rs` — in-flight tracking, cancellation sending
- `mcp/handler.rs` — handle incoming cancellation notification
- `mcp/protocol.rs` — notification builder helper

**Effort:** Medium.

### WI-09: Resource Subscriptions — ✅ IMPLEMENTED

**What the spec says:** Clients can `resources/subscribe` to a URI and receive
`notifications/resources/updated` when it changes.

**Current state:** We receive `notifications/resources/updated` and emit an Info
event. But we never subscribe.

**Implementation:**

1. **New McpOperation variants:**
   ```rust
   McpOperation::SubscribeResource { uri: String },
   McpOperation::UnsubscribeResource { uri: String },
   ```

2. **Protocol mapping:**
   ```rust
   SubscribeResource { uri } => JsonRpcRequest::new(id, "resources/subscribe", Some(json!({ "uri": uri }))),
   UnsubscribeResource { uri } => JsonRpcRequest::new(id, "resources/unsubscribe", Some(json!({ "uri": uri }))),
   ```

3. **CLI commands** — Add to the dynamic CLI:
   - `subscribe <uri>` — calls SubscribeResource
   - `unsubscribe <uri>` — calls UnsubscribeResource

4. **Result mapping** — Both return empty success. Map to a simple confirmation
   `McpOperationResult::Info { message }` or reuse Pong.

5. **Subscription tracking** — Optional: store active subscriptions in
   StateStore so we can show them and clean up on exit.

6. **Auto-refresh on update** — When `notifications/resources/updated` arrives
   and we have the resource in cache, mark it stale. Already done via
   `handle_resource_updated()`.

**Files touched:**
- `mcp/model.rs` — new operations
- `mcp/protocol.rs` — request mapping
- `mcp/client.rs` — response mapping
- `apps/dynamic.rs` — subscribe/unsubscribe commands

### WI-10: Roots Capability — ✅ IMPLEMENTED

**What the spec says:** Clients that support roots declare
`roots: { listChanged: true }`. The server can then send `roots/list` requests
and the client returns a list of `{ uri, name }` pairs.

**Why it matters for our CLI:** Servers use roots to understand what files/URIs
they should be working with. Without it, a server might refuse to operate or
work with the wrong context.

**Implementation:**

1. **Config** — Add to `AppConfig` (or `ServerConfig`):
   ```yaml
   roots:
     - uri: "file:///home/user/project"
       name: "workspace"
   ```
   In config/mod.rs:
   ```rust
   #[derive(Debug, Clone, Serialize, Deserialize, PartialEq, Eq)]
   pub struct RootEntry {
       pub uri: String,
       pub name: Option<String>,
   }
   ```
   Add `roots: Vec<RootEntry>` to `AppConfig`.

2. **Capability declaration** — In `ProtocolEngine::new()`:
   ```rust
   roots: if has_roots { Some(CapabilityMarker {}) } else { None },
   ```
   Thread the roots config into ProtocolEngine, or just always declare it.

3. **Handler** — In `OperationMessageHandler::handle_request()`, add:
   ```rust
   "roots/list" => handle_roots_list(request, &self.roots),
   ```
   Where `handle_roots_list` returns:
   ```json
   { "roots": [{ "uri": "file:///...", "name": "workspace" }] }
   ```

4. **Thread roots into handler** — Add `roots: Vec<RootEntry>` to
   `OperationMessageHandler`. Populate from config.

5. **`notifications/roots/list_changed`** — If roots are reconfigured at runtime.
   Not needed for V1 — roots come from static config.

**Files touched:**
- `config/mod.rs` — RootEntry, roots field
- `mcp/protocol.rs` — capability declaration
- `mcp/handler.rs` — roots/list handler
- `mcp/client.rs` — thread roots into handler

**Effort:** Small.

### WI-11: Progress Token Sending — ✅ IMPLEMENTED

**What the spec says:** "When a party wants to receive progress updates for a
request, it includes a `progressToken` in the request metadata."

**Current state:** We receive progress notifications fine but never send a
`progressToken`, so servers can't target progress to our specific requests.

**Implementation:**

1. **Generate token** — In `map_operation_to_request()`, wrap the request params
   to include `_meta.progressToken`:

## Nuances & Design Decisions

### 1. Where does task polling live?

The poll loop for foreground tasks belongs in `execute_manifest_command()` in
`dynamic.rs`, NOT in the client layer. The client is a dumb transport — it sends
one request and gets one response. The orchestration (poll, retry, display) is
the app layer's job. This is consistent with how the CLI already works: the app
layer decides what operations to send.

### 2. How does `--background` interact with tasks?

Today `--background` is a CLI-only flag that means "don't wait for the response."
With MCP tasks, it means "use task augmentation AND return immediately after
`CreateTaskResult`." The task ID is stored in `StateStore` and the user can
`jobs wait <id>` later. This is a natural mapping.

For tools with `taskSupport: "required"`, every invocation is task-augmented.
In foreground mode, we poll to completion. In `--background` mode, we return the
task ID immediately.

For tools with `taskSupport: "forbidden"` (default), `--background` is a
client-side-only concept — we just don't wait for stdout but the server processes
synchronously. This is the existing behavior.

### 3. How do we handle version negotiation gracefully?

When we send `2025-11-25` and the server responds `2025-03-26`:
- `session.protocol_version` becomes `"2025-03-26"`.
- When parsing responses, we skip structured content, resource links, etc.
  They simply won't be present.
- In the HTTP transport, we send `MCP-Protocol-Version: 2025-03-26`.

When the server responds `2025-06-18`:
- We get structured content and resource links but NOT tasks.

No special case logic needed — the features are present in responses or they're
not. The version is informational. We parse what exists.

### 4. Non-interactive elicitation

For CI/CD pipelines (`--non-interactive`), ALL elicitation requests (form + URL)
return `action: "cancel"`. The server handles this gracefully — it's a defined
response action. For sampling, we return the existing error response.

We thread `non_interactive` as a field on `OperationMessageHandler`. It's set
from the global `--non-interactive` flag.

### 5. URL elicitation browser opening

The spec says "MUST NOT open the URL without explicit consent." Our CLI flow:
1. Print the URL and message.
2. Ask `Open in browser? [Y/n/cancel]`.
3. On `Y`: attempt `xdg-open` / `open` / `start`.
4. If browser open fails, print "Please open manually."
5. Wait for Enter to confirm the user completed the interaction.
6. Return `action: "accept"`.

This satisfies all MUST requirements: user sees the URL, gives explicit consent,
we don't pre-fetch.

### 6. `_meta` everywhere

The spec says `_meta` should be on many message types. We only need to:
- Send `_meta.progressToken` on requests (WI-11).
- Send `_meta.io.modelcontextprotocol/related-task` on task-related messages (WI-07).
- Preserve `_meta` from responses when passing data through.

We don't need to modify `_meta` on every message type — just the ones where
protocol semantics require it.

### 7. `into_result()` error handling for -32042

`JsonRpcResponse::into_result()` currently returns a generic `anyhow!` error for
any JSON-RPC error. To handle `URLElicitationRequiredError`:

```rust
if let Some(error) = self.error {
    if error.code == -32042 {
        // Extract elicitations from error.data and display
        // Return a typed error that the caller can match on
    }
    return Err(anyhow!("json-rpc error {}: {}", error.code, error.message));
}
```

For V1, just display the embedded URLs and return the error. The user retries
manually. Auto-retry is a V2 feature.

### 8. Capability advertisement timing

We advertise ALL capabilities we support, regardless of config. Even if
`roots: []` is empty, we declare the `roots` capability — the server sends
`roots/list`, we return an empty list. This is correct per spec.

Similarly, we always declare `elicitation: { form: {}, url: {} }` — we support
both modes even if we're in non-interactive mode (we'll just return "cancel").

### 9. Audio content in a terminal

For `type: "audio"` content in tool results, we can't play audio in a terminal.
Our approach:
- Decode base64 → save to temp file under `$TMPDIR/mcp2cli-audio-<uuid>.<ext>`.
- Print the file path.
- With `--open`, attempt to launch the system audio player.

Same for images — save to temp file, print path, optional `--open`.

### 10. Task `input_required` status

When a task reaches `input_required`, the server has a pending
elicitation/sampling request that needs to be received via `tasks/result`. Our
poll loop detects this status and calls `tasks/result`, which blocks until the
server sends the request. The response is a JSON-RPC request (elicitation or
sampling) that we handle via the normal handler, then the task resumes.

This is the most complex aspect of the task system. For V1, implement the basic
detection and call to `tasks/result`. The handler infrastructure already handles
elicitation and sampling — it's just that the delivery mechanism is different
(via task result instead of inline during a pending request).
